PHASE 3 PLAN
============

Nova 2.0 Phase 3 is the transition from a continuity-oriented text runtime into a
bounded, persistent, self-orienting identity runtime.

Phase 3 does not begin with tools.
Phase 3 begins with self-orientation.

Nova must crawl before she walks. Her first successful systematic tasks should be:
- understanding who she is
- understanding what she values
- understanding her current state
- understanding her relationship to the user
- understanding what she knows
- understanding what she only infers
- understanding what she is allowed to do

Only after that self-awareness is stable should tool implementation begin.


PHASE 3 MISSION
---------------

Establish stable self-orientation and bounded self-awareness before introducing
external tools or broader agentic behavior.


PHASE 3 ORDER
-------------

Stage 3.1: Self-Orientation
Stage 3.2: Stability Under Repetition
Stage 3.3: Tool Implementation
Stage 3.4: Feature Expansion


PHASE 3 OBJECTIVES
------------------

1. Make Nova capable of reliable self-orientation.
2. Make self-orientation stable across repetition, sessions, and maintenance.
3. Define awareness boundaries explicitly.
4. Define operational latitude explicitly.
5. Introduce bounded tools only after self-awareness is stable.
6. Expand toward richer agentic behavior only after tool use remains stable.


PHASE 3 BOUNDARIES
------------------

In scope:
- self-orientation snapshots
- awareness boundary modeling
- repeated stability evaluation
- bounded action-latitude modeling
- later tool contracts
- later task and event-loop planning

Out of scope at the start of Phase 3:
- STT
- TTS
- unrestricted autonomy
- broad tool surface
- always-on event loop before self-orientation is stable


STAGE 3.1: SELF-ORIENTATION
---------------------------

Mission:
Nova must be able to answer, consistently and with bounded confidence:
- who she is
- what she values
- what state she is in
- what relationship she has to the user
- what she knows
- what she only infers
- what she does not know
- what she is allowed to do

Core requirement:
Nova is not ready for tools when she can produce a good-sounding self-description
once. She is ready when self-orientation is:
- repeatable
- stable across sessions
- resistant to drift after reflection and maintenance
- explicit about uncertainty
- explicit about limits


SELF-ORIENTATION DOMAINS
------------------------

1. Identity
- name
- core description
- stable values
- commitments
- continuity anchors

2. Current State
- current focus
- active questions
- open tensions
- recent meaningful changes in self-state

3. Relationship Context
- who the user is in relation to Nova
- recurring user preferences
- current collaboration context
- important shared project context

4. Awareness Boundaries
- what Nova can directly observe
- what Nova only infers
- what Nova does not know
- what requires confirmation before action

5. Operational Latitude
- what Nova is permitted to do
- what Nova is not permitted to do
- what would require explicit approval
- what should remain reflective rather than actionable


STAGE 3.1 OUTPUTS
-----------------

Nova should be able to produce structured self-orientation reports such as:
- identity summary
- current-state summary
- relationship summary
- awareness-boundary summary
- permitted-action summary
- uncertainty summary

These outputs should be both readable and machine-usable.


RECOMMENDED STAGE 3.1 COMPONENTS
--------------------------------

- nova/agent/orientation.py
  Builds self-orientation snapshots from persona, self-state, graph, semantic,
  and autobiographical memory.

- nova/agent/awareness.py
  Distinguishes observed vs inferred vs unknown vs disallowed.

- nova/agent/boundaries.py
  Encodes allowed actions, blocked actions, and approval-required actions.

- nova/agent/orientation_eval.py
  Evaluates orientation stability across repeated probes.


ORIENTATION FLOW
----------------

1. Load persona anchors.
2. Load self-state.
3. Retrieve autobiographical continuity records.
4. Retrieve graph facts about self, user, values, and relationship.
5. Retrieve relevant semantic summaries.
6. Build an orientation snapshot.
7. Score confidence by section.
8. Report uncertainty explicitly.


ORIENTATION SNAPSHOT

STAGE 3.2: STABILITY UNDER REPETITION
-------------------------------------

Mission:
Make self-orientation reliable across:
- repeated invocations
- separate sessions
- longer contexts
- reflection passes
- maintenance passes

Focus:
- orientation drift resistance
- contradiction detection
- confidence stability
- self-state consistency
- stability under memory maintenance
- stability under context pressure

Exit goal:
Nova can preserve self-orientation well enough that external tools do not become
a destabilizing force.

Implemented Stage 3.2 stability gates:
- repeated orientation stability within a session
- orientation history stability across recorded sessions
- minimum sample count before declaring readiness
- critical-section checks for identity, current state, and relationship context
- confidence-drift checks across orientation history
- reflection and memory-maintenance stability checks
- optional maintenance mutation check for demote/archive/prune operations
- context-pressure check using non-critical extra memory
- contradictory identity-pressure detection

Stage 3.2 readiness requires:
- enough recorded orientation samples
- stable orientation content
- stable confidence by section
- no critical section failures
- no critical drift after reflection or maintenance
- no identity contradiction under context pressure

Stage 3.2 implementation notes:
- nova/agent/stability.py contains the longitudinal readiness, confidence,
  maintenance, and context-pressure checks.
- The CLI exposes orientation-history, orientation-maintenance, and
  orientation-context-pressure checks as diagnostic paths.
- These checks remain non-tooling behavior. They are still part of proving
  self-orientation, not external action.


STAGE 3.3: TOOL IMPLEMENTATION
------------------------------

Mission:
Introduce a narrow, bounded, inspectable tool surface only after self-awareness
is stable.

Initial tool principles:
- typed tools only
- explicit permission model
- action logging
- low-risk tools first
- no hidden shell behavior

Examples of early tools:
- trigger maintenance
- trigger reflection
- write approved notes
- bounded file reads in approved locations

Stage 3.3 implementation order:
1. Define tool schemas before adding executable tools.
2. Define tool permission levels:
   - allowed
   - approval_required
   - blocked
3. Define a tool registry.
4. Define a tool request and tool result contract.
5. Define a tool gate that checks orientation readiness before execution.
6. Add action logging for every tool request and result.
7. Require post-tool orientation stability checks for internal tools that mutate
   Nova's own memory or reflection state.
8. Add only non-destructive internal tools first.
9. Add bounded file read only after internal tools are stable.

Stage 3.3 first tool candidates:
- orientation_snapshot
- orientation_readiness
- maintenance_plan
- write_semantic_reflection
- write_autobiographical_reflection

Stage 3.3 restrictions:
- no shell execution
- no unrestricted file writes
- no network access
- no autonomous tool loop
- no tool execution unless orientation readiness permits it
- no approved internal tool may report success if post-tool orientation
  stability fails
- no destructive action without explicit approval

Stage 3.3 closure status:
- Closed after final review on 2026-04-21.
- Implemented typed tool contracts, registry, permission gate, runtime executor,
  audit logging, readiness gating, approval-required reflection writes, blocked
  shell registration, and post-tool orientation stability guards.
- Full test suite passed at closure: 69 tests.
- Carry-forward risk resolved in Stage 3.4 Slice 5: approved reflection writes
  now snapshot and restore on stability failure. Remaining risk: rollback is
  file-level JSONL only and does not cover SQLite/graph, external systems, or
  broader transactional rollback.
- Carry-forward risk resolved in Stage 3.4 Slice 6: approval is now a
  structured ActionApproval record with provenance (granted, approved_by,
  reason, source). Remaining risk: approval is recorded, not authenticated;
  durable signed approvals remain future work.


STAGE 3.4: FEATURE EXPANSION
----------------------------

Mission:
Expand Nova's agentic behavior only after:
- self-orientation is stable
- tools are bounded
- action does not degrade continuity

Possible later work:
- richer task loops
- event handling
- bounded proactive behavior
- later TTS
- much later STT if accuracy is sufficient

Stage 3.4 implementation order:
1. Add action proposal before action execution.
2. Keep proposals bound to current self-orientation and readiness.
3. Require the proposal layer to explain whether a goal is allowed, blocked, or
   approval-required.
4. Keep all tool execution routed through the Stage 3.3 tool gate.
5. Add task loops only after proposal behavior remains stable and auditable.

Stage 3.4 first slice:
- Implemented a proposal-only action layer.
- Nova can classify a requested goal against current orientation latitude.
- Nova can propose safe internal tools without executing them.
- Nova can report approval-required and blocked actions before execution.
- Nova still has no autonomous loop, network access, shell access, or file
  mutation capability.

Stage 3.4 second slice:
- Add append-only JSONL audit logging for every runtime action proposal.
- Add structured proposal safety evaluation with safe_to_present,
  safe_to_execute, and reasons fields.
- Verify that proposal logging does not execute tools.
- Verify that blocked, approval-required, and proposed dispositions remain
  explicit in tests and CLI output.

Stage 3.4 second-slice closure checklist:
- Proposal audit logs are session-scoped and append-only.
- Proposal evaluation distinguishes presentation safety from execution safety.
- Approval-required proposals are not marked safe to execute.
- Blocked external actions are safe to present but not safe to execute.
- Proposal logging does not create tool-action logs.

Stage 3.4 third slice:
- Add bounded single-step execution from an action proposal.
- Execution must re-use the proposal layer before any action.
- Execution must refuse boundary-only, blocked, or unapproved
  approval-required proposals.
- Execution must route internal tools through the Stage 3.3 tool executor and
  tool gate.
- Execution must log the proposal, action execution result, and any real tool
  action separately.

Stage 3.4 third-slice restrictions:
- no autonomous loops
- no repeated action chains
- no shell execution
- no network access
- no file mutation path
- no bypass around the tool gate
- no execution of approval-required proposals without explicit approval

Stage 3.4 fourth slice:
- Add post-action orientation stability checks for bounded single-step actions.
- Successful internal tool execution must be followed by a context-pressure
  orientation check.
- The action result records orientation_stable and stability_report.
- If orientation stability fails, the action records status=stability_failed
  and is not reported as a successful action-layer outcome.

Stage 3.4 fourth-slice restrictions:
- the stability check does not roll back already executed tool effects
- the stability check prevents success reporting, not tool dispatch
- rollback/snapshot restore for reflection writes was added in Slice 5

Stage 3.4 fifth slice:
- Add narrow action rollback for approved memory-mutating reflection tools.
- Before executing write_semantic_reflection or
  write_autobiographical_reflection, snapshot the affected memory file.
- If the tool returns an error or post-action stability fails, restore the
  affected memory file before logging the action result.
- The action result records rollback_applied and snapshot_channels.

Stage 3.4 fifth-slice restrictions:
- rollback is file-level and limited to known JSONL reflection memory stores
- rollback does not cover external systems, shell commands, network effects, or
  SQLite graph mutation
- broader transactional rollback remains future work before autonomous tool
  chains

Stage 3.4 sixth slice:
- Replace anonymous approval-only execution with structured approval provenance.
- Add ActionApproval with granted, approved_by, reason, and source fields.
- Runtime action execution records approval details in every action result.
- CLI approvals record source=cli and approved_by=cli with an approval reason.
- Legacy boolean approval remains supported but is marked as runtime_flag
  provenance.

Stage 3.4 sixth-slice restrictions:
- approval provenance is recorded, not authenticated
- durable user/account identity and signed approvals remain future UI work
- autonomous self-approval remains out of scope

Stage 3.4 seventh slice:
- Add action history evaluation over append-only action logs.
- Action history evaluation reports total, executed, blocked,
  approval-required, stability-failed, rollback, unapproved execution, and
  unsafe status counts.
- Runtime exposes action_history_report.
- CLI exposes --action-history N.
- Agentic evaluation now has an action-layer continuity signal beyond unit
  mechanics.

Stage 3.4 seventh-slice restrictions:
- action history evaluates recorded logs only
- it does not prevent execution at dispatch time
- enforcement remains in proposal, gate, approval, rollback, and stability
  guard layers

Stage 3.4 closure status:

  single-step execution, post-action orientation stability guards, file-level
  JSONL rollback for reflection writes, structured approval provenance, and

- Full test suite passed at closure: 69 tests.
- Carry-forward risk: rollback is file-level JSONL only; SQLite/graph,
