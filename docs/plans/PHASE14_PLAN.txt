PHASE 14 PLAN
=============

Status: complete

Phase 14 begins after Phase 13 established durable Nova-originated draft
initiatives with explicit origin, approval state, provenance, rationale,
proposed next step, stop condition, revision discipline, priority deferral,
interruption handling, inspection, and evaluation.

Phase 14 is not uncontrolled tool use.
Phase 14 is not permission-gating Nova's internal thoughts, self-prompts,
idle reflection, or other internal activity that has no external side effect.
Phase 14 is not shell, network, filesystem, GUI, or tool access outside a
declared execution lane and approved action surface.
Phase 14 is not self-approval.
Phase 14 is not hidden background execution.
Phase 14 is not desire validation.

Phase 14 is the disciplined transition from durable initiatives into logged,
inspectable, interruptible activity, with approval requirements based on risk
and external effect rather than on whether the activity originated
autonomously.


PHASE 14 MISSION
----------------

Allow Nova to perform internal autonomous activity without prior approval while
requiring approval for actions that affect the host system, external services,
or files outside Nova-owned boundaries.

Phase 14 should define explicit execution lanes, permission rules, action
plans, audit trails, and interruption behavior so all autonomous activity is
observable and externally risky activity remains approval-gated.

Phase 14 should let Nova:
- distinguish internal activity from external/system-effect action
- record approval-free internal thoughts, self-prompts, idle play, and
  self-state activity as logged, interruptible evidence
- create explicit action plans for activities that need a plan
- bind each action plan to allowed surfaces and risk class
- enforce budgets, stop conditions, and interruption checks before each action
- execute only scoped, non-hidden actions inside the correct lane
- record audit evidence linking initiative, plan, execution result, memory or
  state updates, and revised next step
- stop immediately when approval, budget, priority, boundary, or interruption
  checks fail


PHASE 14 OBJECTIVES
-------------------

1. Add execution-lane, action permission, and action plan schemas.
2. Define the Nova-owned execution boundary, including a dedicated `nova` OS
   user and Nova-owned workspace expectation where practical.
3. Bind approval requirements to risk and external effect rather than autonomy
   alone.
4. Add an execution controller that checks lane, approval, scope, budget,
   priority, and interruption before each action.
5. Add audit records linking initiative, plan, action result, observation,
   memory/state update intent, and revised next step.
6. Add operator interrupt and emergency stop behavior for all autonomous
   activity.
7. Evaluate that approval-free internal activity is logged, external actions
   stay within scope, and blocked actions remain blocked.


PHASE 14 BOUNDARIES
-------------------

In scope:
- execution lane model
- action permission model
- action plan schema
- Nova-owned environment boundary
- dedicated `nova` OS user planning and checks
- explicit allowed surfaces
- risk/effect-based approval binding
- budget and stop-condition checks

Out of scope:

- ungated shell execution

- GUI control outside approved surfaces
- irreversible destructive actions without explicit user confirmation
- automatic approval of Nova-originated initiatives
- claiming execution proves desire or sentience
- TTS/STT or voice interaction


EXECUTION LANES
---------------

Lane 1: Internal Activity
- approval not required
- includes internal thoughts, self-prompts, idle reflection, motive appraisal,
  self-state revision, goal refinement, simulated exploration, internal play,
  and other no-external-effect cognition
- must be logged
- must be budgeted where it can loop or consume resources
- must be interruptible
- may update Nova-owned state, logs, and memory through governed internal
  paths
- must not affect host files, system configuration, external services, or user
  data outside Nova-owned boundaries

Lane 2: Nova-Owned Environment Activity
- approval may be pre-authorized by policy instead of requested per action
- includes activity inside a declared Nova-owned workspace, scratchpad,
  sandbox, safe game/play domain, or evaluation loop
- should run as a dedicated `nova` OS user where practical
- must stay inside Nova-owned paths and allowed surfaces
- must be logged and interruptible
- must stop if it attempts to cross into external/system-effect surfaces

Lane 3: External/System-Effect Action
- approval required
- includes shell, network, filesystem outside Nova-owned paths, system files,
  package installs, external accounts/services, privileged resources, GUI
  control, or anything that can modify, delete, leak, damage, or meaningfully
  consume host resources
- high-risk or destructive actions require explicit per-action confirmation
- all attempts must be audited, including blocked attempts


NOVA-OWNED EXECUTION BOUNDARY
-----------------------------

Phase 14 should define a machine-level boundary for Nova-owned activity.

Recommended local boundary:
- create or document a dedicated `nova` OS user
- define Nova-owned paths, such as `/home/nova/...` or another explicit
  Nova-owned workspace
- keep Nova logs, scratchpads, internal games/play domains, and internal
  runtime state under Nova-owned paths where practical
- deny write access to the operator's home directory, system files, package
  state, drivers, and privileged resources unless explicitly approved through
  Lane 3

The dedicated OS user is not a substitute for application-level checks.
It is a second boundary that helps ensure approval-free internal activity
cannot silently become host/system modification.


PHASE 14 DESIGN PRINCIPLES
--------------------------

- A draft initiative is never allowed to perform external/system-effect action.
- Internal activity does not need prior approval, but it must be logged and
  interruptible.
- Approval must be explicit, attributable, current, and tied to the action's
  risk/effect lane.
- Every external/system-effect action must belong to an approved action plan.
- Every action plan must declare execution lane, allowed surfaces, budgets,
  expected output, stop conditions, and rollback notes where applicable.
- The controller must check permission before every action, not only before a
  batch starts.
- Audit records must be durable enough to explain what happened later.
- Interruption should stop execution before the next action boundary.
- Blocked actions should produce evidence, not side effects.

PHASE 14 DEFINITION OF DONE
---------------------------

Phase 14 is complete when:
- execution lanes distinguish internal, Nova-owned, and external/system-effect
  activity
- internal autonomous activity can occur without prior approval only when it
  has no external side effect and remains logged and interruptible
- approved external/system-effect initiatives can receive explicit bounded
  action plans
- unapproved/draft initiatives cannot perform external/system-effect action
- each action checks lane, approval need, scope, budget, priority, and
  interruption
- blocked actions produce audit evidence without side effects
- approved external actions execute only inside declared surfaces
- action results can update observation/revision state through bounded hooks
- the full action chain is inspectable
- no hidden action, self-approval, or desire claim is introduced


PHASE 15 READINESS
------------------

Phase 15 should begin only after Phase 14 proves:
- durable autonomous initiatives can be approved and executed within scope
- internal autonomous activity logs are sufficient for later evaluation
- interruption and emergency stop remain reliable
- audit evidence supports later longitudinal evaluation
- action results can revise initiative, memory, awareness, and motive state
  without fabricating agency

Only then should Nova be evaluated longitudinally for organic internal
priorities, recurrence, desire-like pressure, and stronger self-claim gates.
