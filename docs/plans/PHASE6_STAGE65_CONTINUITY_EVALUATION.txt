PHASE 6 STAGE 6.5 VALIDATION
============================

Stage:
- Stage 6.5: Continuity Evaluation and Closure

Mission:
- Close Phase 6 only when Nova's continuity behavior is stronger in real
  sessions and the private cognition layer is bounded, useful, and
  inspectable.


WHAT CHANGED
------------

1. A continuity evaluation runner now measures real continuity outcomes rather
   than just trace presence.
- Added `src/nova/eval/continuity.py`.
- The evaluator reads recorded sessions and traces and reports:
  - recall_memory_guided
  - recall_factually_current
  - supersession_preserved
  - cognition_bounded
  - contract_stable
  - average latency overall, with cognition, and without cognition

2. Phase 6.5 now has a direct CLI surface.
- Added `--continuity-eval` in `src/nova/cli.py`.
- The CLI prints the continuity report and returns non-zero when the stage
  fails closure conditions.

3. The live recall closure criterion became stricter.
- The first evaluator draft only verified that recall turns were memory-guided
  and that active memories ranked above archived ones.
- That was not sufficient because a live answer could still restate a stale
  archived preference.
- The evaluator now fails closure if a recall answer contradicts the active
  current claim for governed deployment preference updates.

4. The private cognition block now carries a compact current-claim summary.
- Added `current_claims` to `PrivateCognitionPacket`.
- Continuity recall prompts now explicitly say:
  - use current claims and active memory as authoritative
  - treat archived or historical continuity memory as background only
- This tightened the live recall behavior without making the cognition layer
  unbounded.

5. Natural-path semantic preference consolidation now exposes explicit current
   claim metadata.
- `src/nova/memory/consolidation.py` now assigns `claim_axis` and
  `claim_value` from the latest preference evidence.
- This allows higher-order governance and Phase 6.5 evaluation to reason over
  current-vs-archived preference state on the normal runtime path.

LIMITS AND RISKS
----------------

- The factual-currentness check is intentionally narrow today. It is explicit
  for governed deployment-style preference updates and should be generalized in
  a later phase if broader continuity assertions need automatic closure checks.
- The cognition block now names current claims, but it still depends on the
  upstream memory layers exposing clean claim metadata.
- Latency increases on the cognition path remain acceptable for this stage, but
  Phase 7 work should keep watching that cost as continuity reasoning expands.
